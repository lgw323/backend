## 1. 프로젝트 개요
---
### 한 줄 정의
CASTMATCH 캐스팅 매칭 플랫폼

### 제품 목표
유튜브 URL 기반 영상 포트폴리오와 작품 크레딧을 표준화해 창작자·배우의 작업 이력을 한곳에 모으고, 축적된 데이터로 감독/작가/제작사의 배우 탐색과 캐스팅 제안을 가능하게 한다.

### 배경
영상 기반 포트폴리오가 개인 채널·SNS·링크 모음에 흩어져 있고, 작품에서 ‘누가 어떤 역할로 참여했는지’가 검색/검증 가능한 형태로 남지 않는다. 독립 제작 환경에서는 캐스팅이 지인/커뮤니티/오픈채팅에 의존해 비효율이 크다. CASTMATCH는 무거운 캐스팅 플랫폼 이전 단계로, 먼저 영상 포트폴리오와 크레딧 데이터를 축적해 이후 캐스팅 제안 MVP로 확장한다(Phase 0→1→2).

### 사용자 문제
1. 영상 포트폴리오가 플랫폼별로 흩어져 한눈에 보여주기 어렵다. 
2.  비전공자는 포트폴리오 페이지를 ‘보기 좋게’ 만들기 어렵고, 만들어도 확산/홍보가 어렵다. 
3.  작품 참여자(배우/감독/스태프) 정보가 구조화되어 연결되지 않는다. 
4. 배우의 출연 이력·배역·장르 경험을 증명/관리하기 어렵다. 
5. 감독/제작자는 캐릭터에 맞는 배우를 효율적으로 찾기 어렵고 제안/응답이 DM에 흩어진다.

&nbsp;

### 해결 방안
Phase 0에서 유튜브 URL로 작품을 빠르게 등록하고(자동 메타데이터 수집+추가 정보 입력), 작품 단위 크레딧을 역할별로 구조화해 참여자 프로필과 연결한다. 개인 공개 포트폴리오 페이지를 자동 생성해 ‘내가 등록한 작품’과 ‘내가 참여한 작품’을 함께 정리한다. Phase 1에서 프로젝트/캐릭터를 생성하고 배우를 검색해 캐릭터 기반 캐스팅 제안을 보내며, 배우는 제안을 수락/거절하고 오디션은 외부 링크로 제출한다(직접 업로드/DM/결제/AI는 후순위).

### 차별점
1. ‘영상 업로드’가 아닌 ‘URL 기반 등록’으로 진입장벽을 낮춤. 
2. 작품 크레딧을 역할별 데이터로 구조화해 사람의 이력(필모그래피)과 자동 연결. 
3. 포트폴리오 허브(Phase 0)→캐스팅 제안 MVP(Phase 1)로 단계적 확장해 초기 개발 부담을 줄이면서 데이터 네트워크 효과를 확보. 
4. 캐스팅 제안을 ‘프로젝트/캐릭터’ 단위로 관리해 DM 분산 문제를 해결.

&nbsp;

### 타겟 사용자
초기 핵심 타깃: 미디어 관련 학과생(졸업작품/과제 영상 정리), 배우 지망생(출연작·연기 영상 공개), 독립영화/웹드라마 감독·작가·제작팀(소규모 캐스팅), 프리랜서 영상 창작자(촬영/편집/음향/음악 포트폴리오). 공통 특성: 작품이 유튜브/드라이브/SNS 등에 분산되어 있고, 참여 크레딧과 협업 이력이 체계적으로 연결되지 않아 ‘발견 가능성’과 ‘신뢰도’가 낮다.

### 사용자 시나리오
[Phase 0] 배우/창작자: 회원가입→프로필 작성→유튜브 URL로 작품 등록→작품 정보 보완(장르/연도/설명/본인 역할)→크레딧 등록(감독/배우/촬영/편집 등)→자동 생성된 공개 포트폴리오 링크를 공유. 

&nbsp;

[Phase 1] 감독/작가/제작팀: 회원가입→프로젝트 생성→캐릭터 생성→배우 검색(나이대/성별/키/지역/장르 경험/특기 등)→배우 상세 확인(출연작/연기영상/포트폴리오)→캐스팅 제안 전송(응답 기한/오디션 링크 요청)→배우의 수락/거절 및 오디션 링크 확인.

### 핵심 지표
Phase 0: 가입 후 7일 내 작품 1개 이상 등록 비율, 작품당 평균 크레딧 등록 수, 공개 포트폴리오 링크 생성/공유 수, 프로필 완성도(필수 항목 입력률). 

Phase 1: 프로젝트 생성 수, 캐릭터 생성 수, 배우 검색→상세 조회 전환율, 캐스팅 제안 발송 수, 제안 응답률(수락/거절), 오디션 링크 제출률, 제안 처리 리드타임.

### 리스크
1. 크레딧 입력의 번거로움으로 데이터가 축적되지 않을 위험.
2.  초기 네트워크 효과 부족(배우/감독 양측 사용자 밀도).
3.  유튜브 메타데이터 변경/제한 등 외부 의존성. 
4. 개인정보/연락처 노출 및 스팸 제안 이슈(연락 가능 여부, 차단/신고 필요). 
5. 저작권/초상권 이슈(영상 링크 공유 범위/공개 설정).

## 1. 회원가입/로그인 및 기본 온보딩
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 시작전

이메일/소셜 기반 회원가입과 로그인, 사용자 유형 선택을 제공해 Phase 0~1의 핵심 흐름(프로필 생성, 작품 등록, 검색/제안)을 시작할 수 있게 한다.

**수용 기준:**
```
1. ○ 사용자는 이메일 회원가입 및 로그인(비밀번호 재설정 포함)을 할 수 있다.
2. ○ 사용자는 소셜 로그인(최소 1종)을 통해 가입/로그인할 수 있다.
3. ○ 온보딩에서 사용자 유형(배우/감독·작가/스태프·창작자 등)을 선택할 수 있다.
```

## 2. 프로필 및 공개 포트폴리오 페이지 생성(Phase 0 핵심)
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 시작전

사용자가 기본 프로필을 작성하면 외부 공유 가능한 공개 포트폴리오 페이지가 자동 생성되며, 대표 작품/내 작품/참여작이 한 곳에 정리된다.

**수용 기준:**
```
1. ○ 사용자는 프로필(이름, 사진, 분야/소개, 연락/링크 등)을 생성·수정할 수 있다.
2. ○ 시스템은 사용자별 공개 포트폴리오 URL을 생성하고 외부에서 조회 가능하다(공개 범위 설정 포함).
3. ○ 포트폴리오 페이지에 대표 작품, 내가 등록한 작품, 내가 참여한 작품이 구분되어 노출된다.
```

#### 2.3.1 대표 작품 설정

서버: PATCH /works/{id}로 is_featured 토글, 사용자당 featured 1개 제약을 트랜잭션으로 보장.

#### 2.3.2 등록작/참여작 자동 구분

서버: 작품 owner_user_id(등록자)와 크레딧 연결 테이블 기준으로 분류 쿼리/집계로 제공.

&nbsp;

**“작품 등록자(owner)”와 “크레딧 연결(credit link)”을 분리**해서 잡는 게 가장 안전해요.

- **등록작(내가 등록한 작품)**: `works.owner_user_id = me`
- **참여작(내가 참여자로 연결된 작품)**: `work_credits.linked_user_id = me` (또는 `work_participants.user_id = me` 같은 연결 테이블)

권장 구현 방향(서버 중심):

1. **데이터 모델**

- `works`: `id`, `owner_user_id`, `visibility`, `created_at` …
- `work_credits`: `id`, `work_id`, `role_code`, `display_name`, `linked_user_id (nullable)`, `created_by`, `created_at` …
  - “미가입자 텍스트”는 `display_name`만 있고 `linked_user_id`는 비움
  - 가입자 연결되면 `linked_user_id` 채움

2. **분류 쿼리(가장 단순/확실)**

- 등록작: `SELECT * FROM works WHERE owner_user_id = :me`
- 참여작: `SELECT DISTINCT w.* FROM works w JOIN work_credits c ON c.work_id = w.id WHERE c.linked_user_id = :me AND w.owner_user_id <> :me`
  - 마지막 조건은 “내가 등록도 했고 참여자로도 연결된 작품”의 **중복 노출 방지**용(원하면 중복 허용도 가능)

3. **중복/우선순위 정책**

- 보통 UX는 **등록작을 우선**으로 두고, 참여작에서는 `owner_user_id != me`만 보여주면 깔끔합니다.
- 반대로 “내 역할이 있는 등록작”을 따로 보여주고 싶으면 `등록작+내역할있음` 같은 세 번째 그룹을 만들 수 있어요.

4. **성능/확장**

- 초기엔 실시간 집계로 충분하고,
- 트래픽이 늘면 `work_user_links(work_id, user_id, relation_type(owner|credited))` 같은 **머티리얼라이즈(전개) 테이블**을 두고 크레딧 연결/해제 이벤트 때 갱신하면 좋아요.
- 인덱스는 최소로 `works(owner_user_id)`, `work_credits(linked_user_id, work_id)` 추천.

## 5. Phase 1 캐스팅 제안 MVP(프로젝트/캐릭터/검색/제안/응답/오디션 링크)
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 시작전

감독·작가·제작팀이 프로젝트와 캐릭터를 생성하고 배우를 검색해 캐스팅 제안을 보낼 수 있으며, 배우는 제안을 수락/거절하고 오디션 영상은 외부 링크로 제출한다. DM/직접 업로드/AI/결제는 제외한다.

**수용 기준:**
```
1. ○ 감독/작가/제작팀은 프로젝트를 생성하고(유형/장르/예정 시기/시놉시스/모집 상태) 관리할 수 있다.
2. ○ 프로젝트 내 캐릭터를 생성하고(나이대/성별/외형 키워드/성격/서사/특기/참고자료/모집 상태) 관리할 수 있다.
3. ○ 감독/작가/제작팀은 배우를 기본 필터(나이대/성별/키/지역/장르 경험/특기/출연작 보유 여부)로 검색할 수 있다.
4. ○ 감독/작가/제작팀은 배우 상세에서 프로필/출연작/연기 영상/포트폴리오 링크를 확인할 수 있다.
5. ○ 감독/작가/제작팀은 프로젝트·캐릭터 기준으로 제안을 발송하고 상태(대기/수락/거절/오디션 제출 완료/마감)를 추적할 수 있다.
6. ○ 배우는 받은 제안을 확인하고 수락/거절(거절 사유 선택 포함)할 수 있다.
7. ○ 오디션 요청 시 배우는 유튜브/비메오/구글드라이브 등 외부 링크를 제출할 수 있다.
```

프로젝트명, 작품 유형, 장르, 제작 예정 시기, 설명/시놉시스, 모집 상태를 입력해 프로젝트를 생성하고 수정/목록화한다.

프로젝트 내 캐릭터(나이대/성별/외형 키워드/성격/서사/특기/참고자료/모집 상태)를 작성하고 관리한다.
